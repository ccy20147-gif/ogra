# Part 0 · Executive Summary

## 一句话定位

**Orga = 多 Agent 协作的"Windows"——一个产品化的 AI Agent 操作系统，让用户像"组队"一样使用 AI。**

## 三个核心价值

```
1. 完整产品级 Web UI
   ├─ 用户友好的聊天界面（不是 CLI / 简单 Dashboard）
   ├─ 多用户多租户（不是单用户 CLI 工具）
   └─ 移动端 / 桌面端 / 浏览器多端访问

2. 多 Agent 协作的可视化
   ├─ 看 5 个 Agent 在群里讨论（不是黑盒）
   ├─ Agent 可以自主招募（自构建组织）
   └─ A2A 消息总线（不是 Kanban 任务卡）

3. Agent 模板市场
   ├─ 社区共享 Agent（不是每个用户从零搭）
   ├─ 一键 Fork（像 GitHub 一样）
   └─ 评分 / 分类 / 商业化
```

## 目标用户

| 优先级 | 用户群 | 占比目标 | 痛点 | Orga 解决 |
|--------|-------|---------|------|----------|
| **Tier-1** | AI 极客 / 独立开发者 | 40% | 想用 Hermes 的能力但不想碰 CLI | 完整 Web UI |
| **Tier-2** | 知识工作者 / PM / 设计师 | 35% | 想用多 Agent 但不懂技术 | 一键 Fork 模板 + 群聊 |
| **Tier-3** | B 端团队 / 企业 | 25% | 需要 RBAC / SSO / 审计 | 企业版 + 多用户 |

## 技术选型（核心决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 前端基座 | **LobeChat fork** | 完整 Web UI（节省 60% UI 工作） |
| Agent 引擎 | **Hermes** | 200+ LLM + 自进化 + 4 层记忆（节省 60% 底层） |
| 业务编排 | **FastAPI（Python）** | AI 生态 + WebSocket + Celery |
| 前端扩展 | **Next.js 15** | LobeChat 已用，复用 |
| 数据库 | **PostgreSQL 16 + pgvector** | 关系 + 向量一体化 |
| 缓存 | **Redis 7** | 会话/消息/限流 |
| 文件存储 | **MinIO** | Agent 上传/下载 |
| 任务队列 | **Celery** | 异步记忆提取/任务调度 |
| LLM 抽象 | **Hermes 内置**（200+ 模型） | 不重复造 LiteLLM |
| Agent 框架 | **Hermes Kanban Swarm + A2A 协议** | Hermes 多 Agent + Orga 增强 |

### 1.2 三个空白市场（Orga 的机会窗口）

**空白 1：多 Agent 协作的"产品化外壳"**

```
现状：
  - Hermes 引擎极强（自进化、4 层记忆、200+ LLM）但仍是 CLI 工具
  - OpenClaw 网关模式很全但太复杂（开发者向）
  - LobeChat 聊天 UI 优秀但没 Agent runtime

空白：
  ⚠️ 没有"中间地带"——把 Agent 引擎包装成 C 端可用的多 Agent 协作产品
  ⚠️ 没有"群聊式"多 Agent UX（用户看着 5 个 Agent 讨论）
  ⚠️ 没有"自构建组织"产品化（Agent 自主招募）

Orga 切入：
  ✅ LobeChat fork（UI）+ Hermes（引擎）= 立刻填上空白
  ✅ 加 A2A 协议 + 群聊 UX + 模板市场 = 稳定护城河
```

**空白 2：Agent 模板市场**

```
现状：
  - Coze 有 Bot 商店（字节主导，闭源）
  - Dify 有模板（但偏开发者）
  - Hermes/LobeChat 完全无

空白：
  ⚠️ 开源世界没有"Agent 模板市场"
  ⚠️ 没有"一键 Fork Agent 模板"的产品级体验
  ⚠️ 没有 Agent 模板的商业化路径

Orga 切入：
  ✅ 模板市场（发布/Fork/评分/评论）
  ✅ 分类（研究/写作/代码/设计/客服）
  ✅ 商业化（优质模板付费）
  ✅ 与自构建组织联动（Agent 知道自己缺什么能力，去市场找）
```

**空白 3：B 端多用户 Agent OS**

```
现状：
  - Coze 主打 B 端但闭源，企业有数据顾虑
  - Dify 有企业版但定价高
  - 开源世界多用户 / RBAC / SSO 缺失

空白：
  ⚠️ 开源 + 多用户 + 付费 的 Agent 平台几乎空白
  ⚠️ 没有"开源 + SaaS"双轨制的成功案例
  ⚠️ 没有"自部署友好"的企业 Agent 平台

Orga 切入：
  ✅ 开源版（社区免费用）
  ✅ SaaS 版（$20/月起）
  ✅ 企业版（$5K-50K/年）
  ✅ RBAC / SSO / 审计
```

## 12. 关键技术挑战

```
┌────────────────────────────────────────────────────────────┐
│  挑战 1: LobeChat 与 Hermes 状态同步                          │
│  ──────────────────────────────────                        │
│  问题:                                                       │
│  - LobeChat 数据库 vs Hermes SQLite 不同步                    │
│  - 用户改了 Agent 配置，Hermes 没更新                          │
│  - Hermes 改了记忆，LobeChat 不知道                           │
│                                                              │
│  解决方案：                                                  │
│  - 单一数据源（Orga PG 是 source of truth）                   │
│  - 变更推送（Orga → Hermes sync）                            │
│  - 定期校验（每小时一次 reconcile）                            │
│  - WebSocket 通知前端刷新                                    │
│                                                              │
├────────────────────────────────────────────────────────────┤
│  挑战 2: 多 Agent 群聊的"循环"风险                            │
│  ────────────────────────────────                          │
│  问题:                                                       │
│  - Agent A 让 Agent B 帮忙，Agent B 又让 Agent A 帮忙        │
│  - 死循环 + 大量 token 消耗                                  │
│                                                              │
│  解决方案：                                                  │
│  - A2A 消息有 max_depth（默认 5 层）                          │
│  - 消息有 parent_message_id（检测环）                          │
│  - 协调 Agent 强制中断（loop detected）                       │
│  - Token 用量超限自动停止                                    │
│                                                              │
├────────────────────────────────────────────────────────────┤
│  挑战 3: 长期记忆的"幻觉"问题                                 │
│  ────────────────────────────────                          │
│  问题:                                                       │
│  - Agent 可能错误提取记忆（"用户喜欢 Python" 实际没说过）      │
│  - 错误记忆污染后续对话                                      │
│                                                              │
│  解决方案：                                                  │
│  - 记忆提取有置信度评分                                      │
│  - 用户可编辑/删除错误记忆（记忆 GUI）                         │
│  - 记忆生效前用户确认（重要记忆）                              │
│  - 记忆可开关（用户禁用）                                    │
│                                                              │
├────────────────────────────────────────────────────────────┤
│  挑战 4: A2A 消息流的"用户可读性"                              │
│  ─────────────────────────────────                          │
│  问题:                                                       │
│  - A2A 消息是 Agent 间技术消息                                │
│  - 用户看不懂（"skill_id": "abc-123"）                       │
│                                                              │
│  解决方案：                                                  │
│  - A2A 消息渲染层（技术 → 自然语言）                          │
│  - 显示 "A 跟 B 说: 帮我分析下数据"                           │
│  - 隐藏技术细节（除非用户点开）                                │
│  - 用户可"折叠/展开"细节                                    │
│                                                              │
├────────────────────────────────────────────────────────────┤
│  挑战 5: 大规模多用户的性能                                    │
│  ────────────────────────────────                          │
│  问题:                                                       │
│  - 10 万用户，每个 5 个 Agent，每个 Agent 实时流式响应         │
│  - WebSocket 连接数爆炸（50 万 +）                            │
│  - LLM API 限流                                            │
│                                                              │
│  解决方案：                                                  │
│  - WebSocket 集群（Redis Pub/Sub）                           │
│  - LLM API 多账号 + 池化（多 Key 轮询）                       │
│  - 消息队列削峰（RabbitMQ / Kafka）                            │
│  - 限流 + 降级（按用户等级）                                  │
│  - Hermes 多实例（水平扩展）                                  │
│                                                              │
└────────────────────────────────────────────────────────────┘
```

### 16.2 设计原则

```
┌────────────────────────────────────────────────────────────┐
│  原则 1: "清晰优先"（Clarity First）                        │
│  ──────────────────────────────────                        │
│  - 信息架构清晰（用户永远知道下一步）                          │
│  - 状态可见（Agent 在干什么，用户看得到）                      │
│  - 反馈及时（每个操作有响应）                                  │
│                                                              │
│  原则 2: "减少认知"（Reduce Cognitive Load）                  │
│  ────────────────────────────────────                      │
│  - 复杂任务拆解为简单步骤                                    │
│  - 默认值智能（80% 场景不需配置）                              │
│  - 渐进式披露（高级功能隐藏在"展开"中）                        │
│                                                              │
│  原则 3: "可掌控感"（User in Control）                       │
│  ────────────────────────────────────                      │
│  - 任何 Agent 行为可中断                                    │
│  - 任何记忆可编辑/删除                                       │
│  - 任何 Agent 可删除                                        │
│  - 任何自动功能可关闭                                        │
│                                                              │
│  原则 4: "美但不炫耀"（Beautiful but not Showy）            │
│  ──────────────────────────────────────                    │
│  - 留白充足（不挤）                                          │
│  - 配色克制（一个主色调）                                    │
│  - 动效优雅（不炫技）                                        │
│  - 字体考究（衬线大标 + 无衬线正文）                          │
│                                                              │
│  原则 5: "包容性"（Inclusive）                                │
│  ──────────────────                                        │
│  - 中英文双语                                                │
│  - 移动端 / 桌面端一致体验                                   │
│  - 弱视/色盲友好                                            │
│  - 新手引导完整                                              │
│                                                              │
└────────────────────────────────────────────────────────────┘
```

### 20.2 多 Agent 群聊协作流程

```
┌────────────────────────────────────────────────────────────┐
│  群聊协作流程                                                │
│                                                              │
│  1. 用户发起任务                                             │
│     ↓                                                        │
│  2. 协调 Agent 接收                                          │

│     ├─ 给 Agent A: "分析需求"                                 │
│     ├─ 给 Agent B: "画 UX"                                   │
│     └─ 给 Agent C: "评估技术"                                │
│     ↓                                                        │
│  4. Agent 并行执行                                           │
│     ├─ Agent A: 需求分析（30s）                               │
│     ├─ Agent B: UX 流程（45s）                                │
│     └─ Agent C: 技术评估（40s）                               │
│     ↓                                                        │
│  5. 协调 Agent 整合                                          │
│     ├─ 收集 3 个 Agent 结果                                  │
│     ├─ 决定是否需要下一轮                                     │
│     └─ 决定是否需要补充信息                                   │
│     ↓                                                        │
│  6. 返回给用户                                               │
│     ├─ 展示整合结果                                          │
│     └─ 提供"反馈"按钮（满意/不满足）                          │
│                                                              │
│  关键交互：                                                  │
│  - 全过程对用户可见（A2A 消息流）                             │
│  - 用户可中途暂停（⏸️ 暂停所有 Agent）                         │
│  - 用户可补充信息（"等等，加上 A2A 协议"）                     │
│  - 用户可强制中断（"算了，不用讨论了"）                        │
│                                                              │
└────────────────────────────────────────────────────────────┘
```

# 总结

Orga 的本质：**多 Agent 协作的"产品化操作系统"**。

**一句话定义**：
> 让"多 Agent 协作"从极客玩具变成人人可用的产品。

**核心战略**：
> LobeChat 做 UI 层 + Hermes 做引擎层 + Orga 做产品化外壳 + 6 大独家差异化。

**关键数字**：
- MVP 6 周（基于双基座）
- 8 周到公测
- 32 周到企业版
- 2027 年 200w+ 用户 / 5000+ 付费
- 2028 年 100w+ 用户 / 50000+ 付费

**核心壁垒**：
1. 时间差（6-12 个月）
2. 用户基础（GitHub 50k+ stars）
3. 模板市场（社区生态）
4. 6 大独家（不可替代）

**最重要的 3 件事**：
1. **快速上线 MVP**（Loop 1 6 周）
2. **找到 100 个种子用户**（KOL 优先）
3. **持续输出内容**（每周一篇 / 一个视频）
