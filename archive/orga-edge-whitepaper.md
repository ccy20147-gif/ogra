# Part 6 · 4 维度任务路由（Orga Edge 灵魂）

## 6.1 4 维度决策框架

```
┌────────────────────────────────────────────────────────────┐
│  维度 1 · 任务复杂度                                          │
│  ──────────────                                            │
│  简单（分类/摘要/翻译/邮件改写）                            │
│  → 本地（7B-13B 模型）                                      │
│                                                              │
│  中等（写作/分析/简单代码/单元测试）                         │
│  → 本地（13B-30B 模型）或 云端（看其他维度）                │
│                                                              │
│  复杂（推理/长文/复杂代码/深度分析）                        │
│  → 云端（70B+ 模型）                                        │
│                                                              │
│  ─────────────────────────────────────────                  │
│                                                              │
│  维度 2 · 数据敏感度                                          │
│  ──────────────                                            │
│  极敏感（财务/医疗/法律/个人隐私/公司代码）                  │
│  → 本地（强制，不出本地）                                   │
│                                                              │
│  中等敏感（用户行为/可识别数据）                             │
│  → 拆分任务（敏感部分本地 + 公开部分云端）                  │
│                                                              │
│  公开（公开数据/通用任务）                                   │
│  → 云端（按其他维度决策）                                    │
│                                                              │
│  ─────────────────────────────────────────                  │
│                                                              │
│  维度 3 · 网络状态                                            │
│  ──────────────                                            │
│  离线/弱网                                                   │
│  → 本地（强制）                                              │
│                                                              │
│  强网                                                         │
│  → 按其他维度决策                                             │
│                                                              │
│  间歇性网络                                                   │
│  → 边缘优先 + 联网后异步同步云端                              │
│                                                              │
│  ─────────────────────────────────────────                  │
│                                                              │
│  维度 4 · 成本/性能                                            │
│  ──────────────                                            │
│  高频低成本任务（日 1000+ 次简单任务）                       │
│  → 本地（节省 API 费用 70-90%）                              │
│                                                              │
│  低频高价值任务（月 < 100 次复杂任务）                       │
│  → 云端（质量优先，按调用付费）                              │
│                                                              │
│  延迟敏感（< 200ms）                                          │
│  → 本地（边缘响应 < 50ms）                                   │
│                                                              │
│  质量敏感（高精度/高可靠性）                                  │
│  → 云端（GPT-5 / Claude 4.5 / Qwen-Max）                    │
└────────────────────────────────────────────────────────────┘
```

## 6.2 路由决策算法

```python
def route_task(task):
    """Orga Edge 4 维度任务路由决策"""
    
    # 维度 1: 任务复杂度
    complexity = analyze_complexity(task)
    # 输出: "simple" / "medium" / "complex"
    
    # 维度 2: 数据敏感度
    sensitivity = analyze_sensitivity(task.data)
    # 输出: "private" / "mixed" / "public"
    
    # 维度 3: 网络状态
    network = check_network_status()
    # 输出: "offline" / "weak" / "strong"
    
    # 维度 4: 成本/性能
    cost_pref = task.cost_budget
    # 输出: "low" / "high"
    latency_req = task.max_latency_ms
    # 输出: 数字（毫秒）
    
    # 决策优先级（从高到低）
    
    # 1. 敏感数据强制本地（最高优先级）
    if sensitivity == "private":
        return "edge"
    
    # 2. 离线强制本地
    if network == "offline":
        return "edge"
    
    # 3. 延迟敏感强制本地
    if latency_req < 200:
        return "edge"
    
    # 4. 复杂任务默认云端
    if complexity == "complex":
        return "cloud"
    
    # 5. 简单任务默认本地
    if complexity == "simple":
        return "edge"
    
    # 6. 中等任务看成本
    if cost_pref == "high":
        return "cloud"
    
    return "edge"  # 默认本地
```

## 6.3 真实场景示例

### 场景 1：个人写作助手（SeedV + Orga Edge）

```
场景：用户在 MacBook（M3）上写公众号

任务 1: "把这封邮件改写得更礼貌"
  决策路径：
  - 复杂度: 简单（短文本改写）→ 倾向本地
  - 敏感度: 私人（用户邮件）→ 强制本地
  - 网络: 强网
  - 延迟: < 100ms → 强制本地
  → 最终: 本地（Qwen 7B Q4，< 500ms）

任务 2: "写一篇 5000 字 AI Agent 行业分析"
  决策路径：
  - 复杂度: 复杂（长文 + 推理）→ 倾向云端
  - 敏感度: 公开 → 不强制
  - 网络: 强网
  - 成本: 低频高价值 → 倾向云端
  → 最终: 云端（Claude 4.5，30 秒）
  → 任务分发：本地 Agent 构造 A2A 消息发到云端 Orga
  → 云端协调 Agent 分配"写作 Agent"+"研究 Agent"协作
  → 流式返回结果到本地 Mac，渲染保存

用户感知: "我在 Mac 上写，AI 帮我写完 5000 字"
实际: 80% 草稿本地 + 20% 深度内容云端
成本: $0.5 (云端 1 次) vs 全本地 ($0，但质量差)
```

# Part 7 · A2A v2 协议（端云协同扩展）

## 7.1 A2A 协议演进

```
A2A v1（Orga 主线 / Loop 2 实现）
  - 内部多 Agent 协作（同进程 / 同集群）
  - 消息模型：id / from / to / task / context / status
  - 用途：Orga 群聊内的多 Agent 协作

A2A v2（Orga Edge / Loop 3 实现）⭐ 新增
  - 跨节点多 Agent 协作（云 + 端 + 多设备）
  - 消息模型：v1 + 路由元数据
  - 用途：端云协同 + 多节点分布式
```

## 7.2 A2A v2 消息模型

```typescript
{
  "id": "msg-uuid",
  "conversation_id": "group-uuid",
  "from_agent_id": "edge-agent-01",
  "to_agent_id": "cloud-agent-05",
  "task": "分析这 1000 行代码",
  "required_capability": "code_review",
  
  // ⭐ 新增：路由元数据
  "routing": {
    "preferred_node_type": "cloud",   // 偏好节点类型
    "fallback_node_type": "edge",     // 备选节点
    "data_sensitivity": "public",     // public / mixed / private
    "max_latency_ms": 5000,            // 最大延迟
    "min_quality": "high",             // 最低质量
    "cost_budget": "low"               // 成本预算
  },
  
  // ⭐ 新增：数据归属
  "data_locality": {
    "input_data_location": "edge",     // 输入数据在哪
    "output_data_location": "edge",    // 输出数据放哪
    "compute_location": "cloud",       // 计算在哪
    "data_split_strategy": "metadata_only"  // 脱敏策略
```

## 7.3 协调 Agent 的端云路由逻辑

```
┌────────────────────────────────────────────────────────────┐
│  协调 Agent（端云协调器）                                    │
│  ────────────────────                                      │
│                                                              │
│  收到任务 →                                                  │
│    1. 分析任务（复杂度 / 敏感度 / 延迟 / 成本）              │
│    2. 查询节点能力（哪些节点在线？哪些模型？）              │
│    3. 4 维度路由决策（按优先级）                             │
│    4. 构造 A2A v2 消息（带路由元数据）                      │
│    5. 发送 + 跟踪                                           │
│    6. 结果返回 + 状态同步                                   │
│                                                              │
│  决策优先级（从高到低）：                                    │
│    1. 敏感数据 → 本地                                       │
│    2. 离线 → 本地                                           │
│    3. 延迟敏感 → 本地                                       │
│    4. 复杂任务 → 云端                                       │
│    5. 简单任务 → 本地                                       │
│    6. 中等任务 → 看成本                                     │
│                                                              │
│  关键设计：                                                  │
│    - 用户感知: 透明（不知道任务去了哪）                      │
│    - 协调器: 每个 Group 都有（云端或边缘）                  │
│    - 分布式协调: 多节点互相备份                              │
│    - 离线降级: 断网时本地处理 + 联网后同步                   │
└────────────────────────────────────────────────────────────┘
```

## 7.4 A2A v2 跟 v1 的关系

```
A2A v1 + A2A v2 不是替代关系，是扩展关系

A2A v1 处理的：
  - 同进程 Agent 协作
  - 简单任务分发
  - 单节点多 Agent

A2A v2 新增：
  - 跨进程 / 跨设备 Agent 协作
  - 智能路由（4 维度决策）
  - 多节点多 Agent
  - 数据本地化
  - 离线降级

Orga 主线：
  - 内部 Agent 用 A2A v1 协作
  - 跨节点时用 A2A v2 路由
  - 用户无感知（统一体验）
```

# Part 12 · 关键技术挑战 + 解决方案

## 12.1 挑战 1：节点发现 + 能力描述

```
问题：
  - 边缘节点怎么被云端发现？
  - 每个节点的能力（模型 / 硬件 / 可用性）怎么描述？

解决方案：
  - 节点注册中心（云端维护）
  - 边缘节点启动时注册（带能力元数据）
  - 定期心跳（健康检查）
  - 能力描述：模型列表 / 显存 / CPU / 性能等级
```

## 12.2 挑战 2：数据本地化

```
问题：
  - 敏感数据怎么确保不出本地？
  - 用户怎么信任 Orga Edge？

解决方案：
  - 强制本地模式（用户可设）
  - 数据脱敏（异常数据脱敏后云端）
  - 审计日志（数据流向透明）
  - 端到端加密（即使数据上云）
  - 商业模式：开源 + 自部署 = 用户可控
```

## 12.3 挑战 3：网络状态检测

```
问题：
  - 怎么实时检测网络状态？
  - 弱网时怎么优雅降级？

解决方案：
  - 健康检查（定期 ping 云端）
  - 指数退避（重连策略）
  - 离线队列（任务暂存本地，联网后同步）
  - 用户可见状态（在线/弱网/离线）
```

## 12.4 挑战 4：状态同步

```
问题：
  - 边缘和云端的记忆怎么同步？
  - 冲突怎么解决？

解决方案：
  - CRDT（Conflict-free Replicated Data Types）
  - 最后写入获胜（Last Write Wins，简单场景）
  - 人工合并（冲突时用户决定）
  - 分层存储（短期本地，长期云端）
```

## 12.5 挑战 5：协议开销

```
问题：
  - 边缘设备资源有限（CPU / 内存 / 网络）
  - A2A 协议开销不能太大

解决方案：
  - 消息压缩（gzip / protobuf）
  - 连接复用（WebSocket 长连接）
  - 批处理（小消息合并发送）
  - 协议版本协商（兼容老设备）
```
